I learned OOP from a Java project, writing code to build a Painter GUI Application with Swing framework for Graphic User Interface(GUI). It is controlled by the mouse, with features including copying and pasting, drawing rectangles, ellipses and circles, filling colors, and redo and undo. And in order to implement the features, I applied OOP principles with several design patterns. Here are some examples:

First according to the functions I want to get, I introduced classes describing shapes of items, commands of orders, states of application, and states of the painting canvas. Then I use Observer Pattern for the painting canvas to describe the movement of the mouse. The mouse event would change the shape list and the canvas would refresh to draw shapes.

I also applied Adapter Pattern to describe different modes of mouse movement, including drawing, selecting, and moving the mouse. The three classes are the implementation of the same anchestor class, MouseAdaptor, and are watched by the same Mouse observer class.

Besides, Command Pattern is used for implementing selecting shapes as a group and moving the shapes as a group. As Groups could act as a whole to be selected, moved, copied, pasted, and deleted, Groups could be treated like IShape. Moreover, groups could be added into another group, which means there could be a list of IShape describing what shapes are inside the group. That structure comes to composite pattern. The group is hierarchical and could contain other groups or shapes, like directories and files.

My significant project with C is to design and implement a sidecar container project for our company, Model Container. This is part of the Nearline Calculating System. It focuses on heartbeating with the server, and on receiving the heartbeat command, it will fork child process, update the information of child process or kill the child process. We introduce it for nearline computation, which means it could fork a nearline service process when the online service process is not receiving requests, and when there is online request, the nearline process should be killed, and there would be no more nearline request routed here. We choose C programming language for its convenient system-call APIs.

To achieve the functions above, I separated the application code into twoparts: Replica Info extractor and Nearline Command Executor. The Replica Info Executor would extract actions and specific information from heartbeat HTTP request from the Operator(This is the master, and my model container is a slave to act according to the request.), and save them into a map. Nearline Command Executor read the map and call Shell scripts by forking child process to do download and start actions, or stop actions.

I designed and built a remote procedure call service with C++ called Model Server. It aims at changing ONNX, Paddle and other formats of deep learning model structure files into an online predicting service. I chose C++ because C++ has better TensorRT APIs, which is used for the predicting part, and could be connected to cuda code easier than other languages.

The model server has three main functions: networking, data processing, and predicting.

For networking, It uses b-rpc to receive requests by protocols over TCP, and sets parameters for congestion control(e.g.the TCP congestion window size) and and fast fails(when the calculation takes too long, it would be discarded).

For data processing, we should translate the data from RPC requests to the format of Tensor, which is the input of the predictors. As our upstream services would send strings like queries, documents, and urls to us, we would tokenize the strings, translate the tokens into float numbers according to a dictionary, and then fill in the tensors.

For predicting, we generates different predicting APIs including TensorRT, PaddlePredictor and OnnxParser, and we unified their different optimization config into simple ones.
